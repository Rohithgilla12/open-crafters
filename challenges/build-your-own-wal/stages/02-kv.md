# Stage 2: An in-memory key-value store

Before durability, correctness: a plain in-memory map with the exact request
semantics pinned down. Every later stage reuses these operations verbatim —
the only thing that changes is what happens *underneath* them.

## Your task

Implement three methods (keys and values are always strings):

**`set`** — store a value.

```
→ {"id": "1", "method": "set", "params": {"key": "fruit", "value": "apple"}}
← {"id": "1", "result": {}}
```

**`get`** — fetch a value.

```
← {"id": "2", "result": {"value": "apple", "found": true}}     (present)
← {"id": "3", "result": {"value": null, "found": false}}        (absent)
```

**`del`** — remove a key.

```
← {"id": "4", "result": {"deleted": true}}     (the key existed)
← {"id": "5", "result": {"deleted": false}}    (it didn't)
```

## Tests

Overwrites return the latest value; deleting a key doesn't disturb its
neighbors; deleting twice reports `deleted: false` the second time; reading
a never-written key reports `found: false`.

## Notes

- A dictionary/hash map behind your connection handler is all this stage
  needs. If connections are handled on separate threads, put a lock around it
  now — the gauntlet at the end will not forgive data races.
